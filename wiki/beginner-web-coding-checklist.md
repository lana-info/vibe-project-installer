# Памятка Новичка Для Вайбкодинга

Это простая инструкция для работы с проектами через Codex или Claude Code.

Главное правило: не проси AI сразу сделать весь продукт. Работай маленькими проверяемыми шагами: сначала описание продукта, потом задачи, потом одна задача за раз.

## Короткий Маршрут

1. Выбери тип проекта.
2. Создай новый проект или настрой существующий.
3. Сделай `PRD.md` - понятное описание продукта.
4. Проверь `PRD.md` свежим взглядом.
5. Сделай `TASKS.md` - список маленьких задач.
6. Возьми одну задачу.
7. Попроси AI сначала сделать план, а не сразу менять код.
8. Разреши реализацию.
9. Проверь результат.
10. Сделай code review.
11. Сделай commit.

## Типы Проектов

Перед созданием проекта выбери, что именно ты делаешь.

- `Website` - обычный сайт в браузере: страницы компании, блог, каталог, публичная документация.
- `Landing` - одна страница для продажи, рекламы или проверки идеи.
- `Mobile app + Web app` - мобильное приложение плюс веб-приложение/сайт и backend/API.
- `Desktop Python app` - приложение, которое запускается на компьютере.
- `Chrome extension` - расширение для Chrome/Firefox.

Галочка `Docs / документация` почти всегда должна быть включена. Она добавляет стартовые документы: `START_HERE`, `PRD`, `TASKS`, `wiki` и `prompts`.

Галочка `Design starter / основы дизайна` нужна, если проект должен выглядеть современно и тебе нужен дизайн-workflow.

Для `Website` и `Landing` можно включать подходящие web-функции, если они реально нужны проекту. `Desktop Python app` и `Chrome extension` обычно оставляй с документацией и дизайном.

План хостинга нужен для `Website`, `Landing` и `Mobile app + Web app`. Для `Desktop Python app` и `Chrome extension` он обычно не нужен на старте.

## Desktop Python App

Если создаешь desktop-приложение, отдельно выбери интерфейс.

- `tkinter` - самое простое окно, почти без красивого дизайна, ничего скачивать не нужно.
- `customtkinter` - простой, но более современный Python-интерфейс.
- `PySide6 / Qt` - профессиональный desktop-интерфейс для серьезного приложения.
- `Flet` - быстрый способ собрать красивый интерфейс, похожий на web app.
- `React + Tailwind + shadcn/ui + Tauri` - красивый вариант для приложения как рабочий кабинет: sidebar, карточки, поиск, аккуратные кнопки.

Блок `Пакеты интерфейса` по умолчанию лучше держать выключенным. Тогда installer создаст список пакетов и команды в README, но не будет сразу скачивать `.venv`, `node_modules` и тяжелые UI-пакеты.

## Зависимости

Зависимости - это программы и библиотеки, без которых проект не запускается.

Частые системные инструменты:

- `Python` - нужен для installer-скриптов и Python-проектов.
- `Node.js` - нужен для большинства web, frontend и Chrome extension проектов.
- `Bun`, `npm`, `pnpm` или `yarn` - нужны для JavaScript-библиотек.
- `Git` - нужен, чтобы скачивать шаблоны, делать commits и отправлять проект на GitHub.
- `PowerShell` или `pwsh` - нужен для PowerShell-скриптов.

Если зависимостей не хватает, проси Codex сначала объяснить, что нужно и зачем.

```text
Проверь, какие зависимости нужны для запуска проекта.
Сначала объясни, что уже установлено, чего не хватает, зачем это нужно и можно ли обойтись без этого.
Потом установи только нужные зависимости проекта и запусти минимальную проверку.
```

## Новый Проект

Используй этот маршрут, если проекта еще нет или ты создаешь его с нуля.

### Что Делать

1. Создай проект через installer.
2. Открой созданную папку в Codex.
3. Открой `START_HERE` или `README.md`.
4. Сделай или заполни `PRD.md`.
5. Проверь `PRD.md` в новом окне.
6. Сделай или обнови `TASKS.md`.
7. Возьми первую маленькую задачу.
8. Реализуй ее.
9. Проверь запуск или тест.
10. Сделай code review.
11. Обнови `TASKS.md`.
12. Сделай commit.

### Промпт 1: Сделать PRD

`PRD.md` - это понятное описание продукта. Оно нужно, чтобы AI не придумывал продукт заново.

```text
Я хочу сделать продукт: <описание>.

Сначала не пиши код.
Задавай мне вопросы по одному с вариантами ответа.
Когда вопросов не останется, создай или обнови PRD.md.
В PRD.md не должно быть размышлений и открытых вопросов: только понятное ТЗ, MVP, ограничения и критерии готовности.
```

### Промпт 2: Проверить PRD

Лучше делать в новом окне или новом thread, чтобы AI посмотрел свежим взглядом.

```text
Сделай read-only review файла PRD.md.

Проверь:
- есть ли открытые вопросы;
- есть ли противоречия;
- не слишком ли большой MVP;
- понятны ли главные пользовательские сценарии.

Если вопросы есть, задавай их по одному с вариантами ответа.
После ответов обнови PRD.md.
```

### Промпт 3: Сделать TASKS

`TASKS.md` - это список маленьких шагов.

```text
Прочитай PRD.md и создай или обнови TASKS.md.

Разбей работу на маленькие задачи в правильном порядке.
У каждой задачи должен быть результат и способ проверки.
Отдельно пометь MVP и то, что можно сделать позже.
Не добавляй то, чего нет в PRD.
```

### Промпт 4: Сделать Одну Задачу

```text
Возьми первую открытую задачу из TASKS.md.

Сначала ничего не меняй.
Найди связанные файлы, риски и edge cases.
Предложи минимальный план.
После согласования реализуй, проверь, сделай review и обнови TASKS.md.
```

### Промпт 5: Code Review

```text
Сделай read-only code review активных изменений.

Ищи только серьезные подтвержденные проблемы:
- регрессии;
- ошибки логики;
- проблемы безопасности;
- потерю данных;
- нехватку важных тестов.

Верни severity, file:line, доказательство, риск, минимальный fix и тест.
```

## Существующий Проект

Используй этот маршрут, если проект уже есть и ты хочешь подготовить его к работе с AI.

### Что Делать

1. Не создавай новый проект поверх старого.
2. В главном окне переключи `Режим` на `Существующий проект`.
3. Убедись, что создан `SETUP_AUDIT.md`.
4. Попроси Codex прочитать audit и объяснить, что уже есть.
5. Сделай или обнови `PRD.md`.
6. Сделай или обнови `TASKS.md`.
7. Работай по одной задаче.

Режим настройки существующего проекта должен делать только безопасные вещи:

- создать `SETUP_AUDIT.md`;
- добавить недостающие docs/prompts/wiki;
- не перезаписывать существующие файлы;
- не устанавливать зависимости автоматически;
- не менять код приложения без отдельной задачи.

### Промпт 1: Audit Existing Project

```text
Это существующий проект для вайбкодинга.

Прочитай SETUP_AUDIT.md, README, package.json/requirements/pyproject, если они есть.
Ничего не устанавливай и не меняй без объяснения.

Сначала коротко объясни:
- что это за проект;
- как он запускается;
- какие части уже есть;
- каких документов не хватает;
- какие зависимости нужны и зачем.
```

### Промпт 2: Обновить PRD

```text
Помоги создать или обновить PRD.md для существующего проекта.

Сначала используй то, что уже видно из README, SETUP_AUDIT.md и структуры проекта.
Потом задай мне недостающие вопросы по одному с вариантами ответа.
После ответов обнови PRD.md.
Не меняй код.
```

### Промпт 3: Обновить TASKS

```text
Прочитай PRD.md, SETUP_AUDIT.md и текущую структуру проекта.

Создай или обнови TASKS.md.
Раздели задачи на:
- что уже есть;
- что нужно довести до рабочего состояния;
- что входит в MVP;
- что отложено на потом.

Каждая задача должна быть маленькой и проверяемой.
```

### Промпт 4: Работать По Одной Задаче

```text
Возьми первую открытую задачу из TASKS.md.

Сначала ничего не меняй.
Найди связанные файлы и риски.
Предложи минимальный план.
После согласования реализуй, проверь, сделай review и обнови TASKS.md.
```

## Как Использовать Prompts Из Проекта

Папка `prompts` - это готовые инструкции для Codex. Не запускай все промпты подряд. Используй нужный промпт в нужный момент.

Обычно:

1. `create-prd` - в начале проекта или когда идея сильно изменилась.
2. `review-prd` - после первого PRD.
3. `create-tasks` - когда PRD уже понятен.
4. `implement-one-task` - перед каждой маленькой задачей.
5. `code-review` - после рабочей правки и перед commit.
6. `security-review` - перед продакшеном или после задач про логин, платежи, файлы, роли и персональные данные.

Главное правило: важные решения не должны оставаться только в чате. После важных ответов обновляй `PRD.md`, `TASKS.md` или `wiki`.

## Git, Commits, GitHub И Push

Git - это история изменений проекта.

Commit - это сохраненная точка в истории.

GitHub - это место, куда проект можно загрузить онлайн.

Push - это отправка твоих commits на GitHub.

Делай commit после каждой небольшой рабочей задачи, когда:

- код запускается;
- проверка или тест прошли;
- Codex сделал review или ты понимаешь, что изменилось;
- `TASKS.md` или wiki обновлены, если решение поменялось.

Перед commit попроси Codex:

```text
Проверь git status и кратко объясни, какие файлы изменены.
Не трогай чужие изменения.
Если все нормально, сделай commit с понятным сообщением.
```

Перед опасными действиями всегда проси проверку:

```text
Покажи git status, текущую ветку и remote.
Ничего не удаляй и не откатывай без моего разрешения.
```

Не проси `git reset --hard`, `git checkout -- .` или удаление файлов, если точно не понимаешь, что будет потеряно.

## Когда Нужны Тесты

Тест - это автоматическая проверка, что важное поведение не сломалось.

Проси тесты, если задача трогает:

- вход в аккаунт;
- права доступа;
- деньги или платежи;
- базу данных;
- важный пользовательский сценарий;
- исправление бага;
- код перед продакшеном.

```text
Сначала предложи, какие тесты нужны для этой задачи.
Не пиши тесты ради количества.
Нужны проверки, которым можно верить.
Если это баг, сначала сделай падающий тест, потом исправление.
```

## Когда Нужен Рефакторинг

Рефакторинг - это уборка кода без изменения поведения.

Его делают не для красоты, а чтобы проект было легче менять дальше.

Нужен, если:

- AI постоянно путается в одном и том же месте;
- маленькая правка ломает другой экран;
- один и тот же код повторяется много раз;
- компонент стал слишком большим;
- стили UI расползлись;
- непонятно, где что менять.

```text
Проведи read-only review для рефакторинга.

Найди только те места, которые реально сложно поддерживать.
Не предлагай косметику.
Для каждого места объясни простыми словами:
- где проблема;
- почему она мешает;
- как убрать риск;
- какой тест нужен перед изменением.

Пока ничего не меняй.
```

## Когда Нужна Security Проверка

Делай перед продакшеном и после задач про логин, платежи, файлы, роли и приватные данные.

```text
Проведи read-only security review.

Проверь:
- логин;
- права доступа;
- можно ли получить чужие данные, поменяв id;
- секреты и .env;
- загрузку файлов;
- платежи и webhooks;
- логи.

Верни только подтвержденные проблемы и минимальный план исправления.
```

## Перед Продакшеном

Продакшен - это когда реальные пользователи будут пользоваться продуктом.

Перед этим проверь:

- [ ] продукт запускается;
- [ ] главные сценарии работают;
- [ ] секреты не в коде;
- [ ] логин и права доступа проверены;
- [ ] важные тесты проходят;
- [ ] есть способ посмотреть ошибки;
- [ ] есть план отката;
- [ ] сделан security review.

```text
Проведи production readiness review.

Проверь, что проект готов к реальным пользователям.
Верни блокеры, риски и самый короткий план подготовки.
```

## Goal Для Длинной Работы

`goal` удобен, когда задача длинная и не хочется каждый раз заново объяснять цель.

Используй `goal`, если:

- нужно несколько шагов подряд;
- есть понятный финальный результат;
- важно не потерять контекст;
- задача может занять больше одного сообщения.

Не используй `goal` для маленькой просьбы вроде "проверь git status" или "переименуй кнопку".

### Goal Для Нового Проекта

```text
Создай goal: подготовить этот новый проект к вайбкодингу до первой маленькой реализуемой задачи.

Сначала прочитай README, START_HERE, PRD.md если есть, TASKS.md если есть.
Если PRD еще нет, помоги мне создать PRD через вопросы по одному.
Потом проверь PRD, создай TASKS.md и предложи первую маленькую задачу.
Не устанавливай зависимости и не меняй код без объяснения зачем.
```

### Goal Для Существующего Проекта

```text
Создай goal: безопасно подготовить существующий проект к вайбкодингу и найти первую маленькую задачу.

Прочитай SETUP_AUDIT.md, README, package.json/requirements/pyproject, если они есть.
Ничего не устанавливай и не меняй без объяснения.
Сначала дай короткий audit: что это за проект, как он запускается, какие зависимости уже описаны, чего не хватает.
Потом предложи, какие docs нужно создать или обновить: PRD.md, TASKS.md, wiki.
```

Goal можно считать завершенным, когда нужный файл создан или обновлен, код проверен, `TASKS.md` отражает новый статус и есть понятный следующий шаг.
