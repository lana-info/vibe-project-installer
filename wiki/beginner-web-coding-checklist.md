# Beginner Web Coding Checklist

Этот файл для новичка. Он объясняет, что просить у Codex/Claude Code, когда это делать и зачем.

## Сначала Выбери Основу

Перед созданием проекта выбери тип продукта.

- `Mobile app + Web` - обычный продукт: отдельное мобильное приложение для телефона, сайт/веб-приложение для браузера на телефоне и ПК, плюс backend/API.
- `Chrome extension` - расширение для Chrome/Firefox. Это отдельная основа, ее не нужно смешивать с mobile/web.

Галочка `Docs / документация` почти всегда должна быть включена. Она добавляет `START_HERE`, `PRD`, `TASKS`, `wiki` и `prompts`, чтобы Codex понимал проект.

Галочка `Design starter / основы дизайна` отдельная. Включай ее, если проект должен выглядеть современно и нужно добавить дизайн-workflow: shadcn/ui Blocks, Magic UI, Origin UI и React Native Reusables как источники идей.

Для `Chrome extension` доступны только документация и `Design starter`. Оплаты, uploads, push, admin, background jobs и другие app-функции там не включаем на старте.


Правильное поведение: Codex должен открыть GUI-настройку, а не ставить все тихо без выбора.

## Самая Простая Схема

1. Сначала описываем продукт.
2. Потом разбиваем работу на задачи.
3. Потом берем одну маленькую задачу.
4. Потом просим AI сделать план.
5. Потом разрешаем менять код.
6. Потом проверяем.
7. Потом просим review.
8. Потом делаем commit.

Не нужно делать весь проект одним запросом.

## Какие Части Проекта Бывают

### webapp

Это закрытая часть сайта после входа в аккаунт: личный кабинет, dashboard, инструменты, админка.

Используй, если пользователь должен залогиниться и что-то делать внутри приложения.

### website

Это публичные страницы: лендинг, блог, SEO-страницы, каталог, pricing, публичная документация.

Используй, если страницу должны находить в Google или красиво открывать по ссылке.

### backend

Это серверная часть: логин, база данных, API, платежи, права доступа, загрузка файлов, интеграции.

Используй почти всегда, если есть аккаунты, данные, платежи или приватная логика.

### mobile

Это отдельное мобильное приложение для телефона.

Используй только если реально нужен App Store / Google Play или функции телефона.

В нашем installer выбор `Mobile app + Web` под капотом включает `web,mobile,backend`, потому что upstream mobile-ветка содержит и приложение, и веб-часть, и backend.

### landing

Это простая продающая страница.

Используй, если пока нужен только первый экран/страница для проверки идеи.

### chrome-extension

Это расширение для браузера.

Используй, если продукт должен работать внутри Chrome/Firefox: popup, страница настроек, content script, background service worker, кнопки на чужих сайтах или автоматизация браузерных действий.

Это отдельная основа проекта. Не смешивай ее с `Mobile app + Web`.

## Новый Проект

### 1. Сделай PRD

PRD - это понятное описание продукта. Оно нужно, чтобы AI не придумывал продукт заново.

Скопируй:

```text
Я хочу сделать продукт: <описание>.

Сначала не пиши код.
Задавай мне вопросы по одному с вариантами ответа.
Когда вопросов не останется, создай PRD.md.
В PRD.md не должно быть размышлений и открытых вопросов: только понятное ТЗ, MVP, ограничения и критерии готовности.
```

Делается всегда перед новым продуктом.

Если проект создан через наш installer, сначала открой `START_HERE.ru.md`, потом `PRD.ru.md`. Не начинай писать код, пока PRD не стал понятным.

### 2. Проверь PRD в новом окне

Новое окно нужно, чтобы AI посмотрел свежим взглядом.

```text
Сделай read-only review файла PRD.md.

Проверь, есть ли открытые вопросы, противоречия или слишком большие куски для MVP.
Если вопросы есть, задавай по одному с вариантами ответа.
После ответов обнови PRD.md.
```

Делается перед тем, как писать задачи.

### 3. Сделай TASKS

TASKS - это список маленьких шагов.

```text
Прочитай PRD.md и создай TASKS.md.

Разбей работу на маленькие задачи в правильном порядке.
У каждой задачи должен быть результат и способ проверки.
Не добавляй то, чего нет в PRD.
```

Делается после PRD.

## Одна Задача

Используй каждый раз, когда хочешь сделать следующий кусок проекта.

```text
Возьми первую открытую задачу из TASKS.md.

Сначала ничего не меняй.
Найди связанные файлы, риски и edge cases.
Предложи минимальный план.
После согласования реализуй, проверь, сделай review и обнови TASKS.md.
```

## Как Использовать Промпты Из Проекта

Папка `prompts` - это рабочие инструкции для Codex. Не нужно запускать все промпты каждый день. Используй нужный промпт перед конкретным этапом.

1. `01-create-prd.md` - один раз в начале проекта или когда идея сильно изменилась.
2. `02-review-prd.md` - после первого PRD, до задач.
3. `03-create-tasks.md` - когда PRD уже понятен.
4. `04-implement-one-task.md` - перед каждой небольшой задачей из `TASKS.md`.
5. `05-code-review.md` - после рабочей правки и перед коммитом.
6. `06-security-review.md` - перед продакшеном, оплатами, логином, загрузками файлов или персональными данными.

Если включены feature packs, используй их промпты по ситуации: payments, uploads-media, e2e-tests, design-starter, background-jobs, scheduled-tasks.

Для `Chrome extension` на старте доступны только `Docs / документация` и `Design starter`. Остальные feature packs не включай, пока проект не станет сложнее и не появится отдельная причина.

Главное правило: решения не должны оставаться только в чате. После важных ответов обновляй `PRD`, `TASKS` или `wiki`.

## Git, Commits, GitHub И Push

Git - это история изменений проекта. Commit - это сохраненная точка в истории. GitHub - это место, куда проект загружается онлайн. Push - это отправка твоих commit-ов на GitHub.

Делай commit после каждой небольшой рабочей задачи, когда:

- код запускается;
- проверка или тест прошли;
- Codex сделал review или ты понимаешь, что изменила;
- `TASKS.md` или wiki обновлены, если решение поменялось.

Не жди, пока весь проект будет готов. Маленькие commits легче откатить и легче понять.

Перед commit попроси Codex:

```text
Проверь git status и кратко объясни, какие файлы изменены.
Не трогай чужие изменения.
Если все нормально, сделай commit с понятным сообщением.
```

Хорошее сообщение commit:

```text
Add Chrome extension project mode
```

Плохое сообщение commit:

```text
fix
```

Push нужен, чтобы изменения попали на GitHub и не остались только на компьютере.

Проси push после commit:

```text
Сделай push текущей ветки в GitHub и скажи commit hash.
```

Если Codex говорит, что `origin` не настроен, значит проект еще не подключен к GitHub-репозиторию. Тогда нужно сначала создать или выбрать репозиторий на GitHub и добавить remote.

Перед опасными действиями всегда проси проверку:

```text
Покажи git status, текущую ветку и remote. Ничего не удаляй и не откатывай без моего разрешения.
```

Никогда не проси `git reset --hard`, `git checkout -- .` или удаление файлов, если точно не понимаешь, что будет потеряно.

## Когда Нужны Тесты

Тест - это автоматическая проверка, что важное поведение не сломалось.

Проси тесты, если задача трогает:

- вход в аккаунт;
- права доступа;
- деньги или платежи;
- базу данных;
- важный пользовательский сценарий;
- исправление бага;
- код перед продакшеном.

Промпт:

```text
Сначала предложи, какие тесты нужны для этой задачи.
Не пиши тесты ради количества.
Нужны проверки, которым можно верить.
Если это баг, сначала сделай падающий тест, потом исправление.
```

## Когда Нужен Рефакторинг

Рефакторинг - это уборка кода без изменения поведения.

Его делают не для красоты, а чтобы проект было легче менять дальше.

Нужен, если:

- AI постоянно путается в одном и том же месте;
- маленькая правка ломает другой экран;
- один и тот же код повторяется много раз;
- компонент стал слишком большим;
- стили UI расползлись;
- никто уже не понимает, где что менять.

Промпт:

```text
Проведи read-only review для рефакторинга.

Найди только те места, которые реально сложно поддерживать.
Не предлагай косметику.
Для каждого места объясни простыми словами:
- где проблема;
- почему она мешает;
- как убрать риск;
- какой тест нужен перед изменением.

Пока ничего не меняй.
```

## Когда Нужен Code Review

Code review - это проверка изменений другим взглядом.

Делай после каждой важной задачи.

```text
Сделай read-only code review активных изменений.

Ищи только серьезные подтвержденные проблемы:
- регрессии;
- ошибки логики;
- проблемы безопасности;
- потерю данных;
- нехватку важных тестов.

Верни severity, file:line, доказательство, риск, минимальный fix и тест.
```

## Когда Нужна Security Проверка

Делай перед продакшеном и после задач про логин, платежи, файлы, роли и приватные данные.

```text
Проведи read-only security review.

Проверь:
- логин;
- права доступа;
- можно ли получить чужие данные, поменяв id;
- секреты и .env;
- загрузку файлов;
- платежи и webhooks;
- логи.

Верни только подтвержденные проблемы и минимальный план исправления.
```

## Перед Продакшеном

Продакшен - это когда реальные пользователи будут пользоваться продуктом.

Перед этим проверь:

- [ ] продукт запускается;
- [ ] главные сценарии работают;
- [ ] секреты не в коде;
- [ ] логин и права доступа проверены;
- [ ] важные тесты проходят;
- [ ] есть способ посмотреть ошибки;
- [ ] есть план отката;
- [ ] сделан security review.

Промпт:

```text
Проведи production readiness review.

Проверь, что проект готов к реальным пользователям.
Верни блокеры, риски и самый короткий план подготовки.
```

## Что Можно Автоматизировать

Самое простое:

- создание `PRD.md`;
- создание `TASKS.md`;
- создание `wiki/daily-workflow.md`;
- создание `wiki/testing.md`;
- создание `wiki/security.md`;
- создание папки `prompts/` с готовыми промптами;
- smoke-check после установки проекта.

В этом installer-проекте стартовые файлы лучше создавать через GUI:

```powershell
python D:\WorkOS\vibe-project-installer\scripts\create_vibe_project_gui.py
```

Для существующего проекта:

```powershell
python D:\WorkOS\vibe-project-installer\scripts\install_project_pack_gui.py --target-path "<папка проекта>"
```

Если документация не нужна, можно отключить галочку docs или использовать `-SkipWorkflowDocs` / `--skip-workflow-docs`, но для обычного vibe-coding лучше оставлять документацию включенной.
