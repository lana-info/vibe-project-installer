# Web Coding Playbook

Версия: 2026-07-06.

Единый рабочий документ по веб-кодингу с AI-агентами на основе альманаха Сухарева / ЙЙ, связанных Telegram-постов, `di-sukharev/vibe`, `github/spec-kit` и программы курса.

Это не архив всех текстов дословно. Это прикладная инструкция: что делать, какие документы держать в проекте, какие промпты давать агенту, когда открывать новый контекст, что автоматизировать и как проверять результат.

## Короткий Вывод

Рабочий цикл такой:

1. Не начинать с кода. Сначала сделать `PRD.md`.
2. Проверить `PRD.md` в новом чистом контексте.
3. Разбить на `TASKS.md`.
4. Делать одну задачу за раз.
5. Перед задачей обсуждать риски и edge cases.
6. После задачи запускать code review.
7. Для сложного поведения писать тесты, лучше e2e/integration.
8. После рабочего куска делать коммит.
9. Перед продакшеном делать security audit и нагрузочную проверку.
10. В проде мониторить ошибки отдельно от исправления root cause.

## Главные Принципы

### 1. Обычного Codex или Claude Code часто достаточно

На старте не надо обвешивать проект MCP, RAG, графами, сложной памятью и десятком скиллов. Сначала агент должен уметь читать код, искать по файлам, запускать команды и делать маленькие изменения.

Когда один и тот же промпт используется постоянно, его можно оформить в skill: например code review, care refactoring, UI refactoring.

### 2. Код и точный поиск важнее абстрактной памяти

Для кода главный источник правды - сам репозиторий. Агент должен:

- смотреть структуру проекта;
- искать точечно через `rg`;
- открывать небольшие участки файлов;
- находить владельца поведения;
- проверять связанные контракты;
- только потом менять код.

RAG, Obsidian, graph-инструменты могут помогать по заметкам, видео, картинкам и архивам, но для кода они не заменяют точный поиск.

### 3. Документация должна быть короткой и продуктовой

В каждом веб-проекте минимально нужны:

- `README.md` - что продукт делает, какую проблему решает, как запустить и проверить.
- `PRD.md` - продуктовые требования без открытых вопросов.
- `TASKS.md` - этапы и задачи.
- `AGENTS.md` или `CLAUDE.md` - правила для агента в этом проекте.
- `wiki/` или `docs/` - долговременная база знаний, решения и runbooks.

Не надо превращать README в пересказ всего кода.

## Стандартная Структура Проекта

Для нового веб-продукта:

```text
project/
  README.md
  PRD.md
  TASKS.md
  AGENTS.md
  wiki/
    decisions.md
    operations.md
    testing.md
    security.md
  errors/
  notes/
  tasks/
```

Для `vibe`-подобного full-stack проекта:

```text
project/
  backend/
  webapp/
  website/
  mobile/
  packages/contracts/
  docs/
  .env.example
  docker-compose.yml
```

Разделение browser surfaces:

- `webapp` - закрытое приложение за логином, CSR, нет SEO.
- `website` - публичные SEO-страницы, лендинги, каталог, контент, SSR/SSG.
- `backend` - API, auth, persistence, jobs, integrations.
- `mobile` - только если реально нужен mobile app; не тащить mobile-логику в веб-проект без причины.

## Новый Продукт: Пошаговый Workflow

### Шаг 1. PRD-интервью

Открой новый проект или пустую папку и дай агенту промпт:

```text
Давай сделаем PRD для продукта <название>.

Идея: <кратко опиши идею>.
Проблема пользователя: <если знаешь>.
Целевая аудитория: <если знаешь>.

Задавай мне вопросы по одному. В каждом вопросе давай 2-4 варианта ответа и возможность ответить свободно.
Я буду отвечать.

Когда вопросов не останется, создай PRD.md в корне проекта.
В PRD.md не должно быть размышлений, TODO и открытых вопросов.
Нужны только четкие продуктовые требования, MVP, ограничения и критерии готовности.
```

Если идея пока сырая:

```text
Я хочу сделать <тип продукта>, но идея еще не до конца сформулирована.
Помоги мне сузить продукт до MVP.
Задавай вопросы по одному с вариантами ответа.
После интервью создай PRD.md без открытых вопросов.
```

### Шаг 2. Независимая проверка PRD

Открой новое окно или новый Codex thread с чистым контекстом:

```text
Сделай read-only review файла PRD.md.

Проверь:
- есть ли открытые вопросы;
- есть ли противоречия;
- хватает ли информации для разработки MVP;
- не смешаны ли MVP и будущие этапы;
- понятны ли пользовательские сценарии и критерии готовности.

Если вопросы есть, задавай их мне по одному с вариантами ответа.
После ответов обнови PRD.md, сохранив формат.
Если вопросов нет, напиши коротко: PRD готов к разбиению на задачи.
```

### Шаг 3. Разбивка на задачи

В новом контексте:

```text
Прочитай PRD.md и разбей работу на этапы и задачи.

Создай TASKS.md в корне проекта.

Требования к TASKS.md:
- задачи должны идти в порядке реализации;
- каждая задача должна давать проверяемый результат;
- отдельно пометь MVP и post-MVP;
- укажи зависимости между задачами;
- для рискованных задач добавь проверку или тест;
- не добавляй задачи, которых нет в PRD.
```

Формат задач:

```markdown
# Tasks

## Phase 1: Foundation

- [ ] T001: ...
  - Result: ...
  - Verify: ...
  - Depends on: ...

## Phase 2: MVP Flow
```

### Шаг 4. Выбор стека и шаблона

Выбирать стек нужно после PRD, а не до него.

Базовые рекомендации из источников:

- backend: TypeScript + Hono + Bun;
- web app: React + Vite для закрытого CSR-приложения;
- public site: Astro или другой SSR/SSG инструмент для SEO;
- mobile: Expo + React Native;
- database: Managed Postgres;
- ORM/schema: Prisma или Drizzle по стеку;
- storage: cloud object storage;
- early production: монолит, без микросервисов и Kubernetes.

Для быстрого старта можно использовать `di-sukharev/vibe`:

```text
Установи шаблон https://github.com/di-sukharev/vibe в этот проект.

Мы делаем <описание продукта>.
Сначала прочитай README.md шаблона и задай короткий intake:
- название проекта;
- активные surfaces: webapp, website, backend/API, mobile или full-stack;
- нужен ли auth;
- нужна ли база данных;
- нужны ли uploads/media;
- нужны ли payments;
- нужны ли admin tools;
- нужен ли deployment сейчас.

После setup зафиксируй bootstrap plan в README.md.
Если это новый проект, не оставляй origin на upstream template.
```

Если нужен mobile:

```text
Используй mobile-ветку шаблона:
https://github.com/di-sukharev/vibe/tree/mobile

Мы делаем мобильное приложение <описание>.
Сначала проверь, что checkout на mobile branch.
Expo/EAS, push, payments и social auth настраивай только если они нужны в MVP.
```

### Шаг 5. Реализация одной задачи

Перед каждой задачей:

```text
Возьми первую открытую задачу из TASKS.md.

Сначала ничего не меняй.
Прочитай связанные файлы и найди владельца поведения.
Затем опиши:
- какие есть edge cases;
- какие есть риски;
- какие части системы будут затронуты;
- какие варианты реализации есть;
- какой минимальный вариант ты рекомендуешь.

Я отвечу с продуктовой точки зрения.
После согласования составь план реализации и выполни его.
```

После согласования:

```text
Реализуй согласованный план минимальным достаточным изменением.

Правила:
- не трогай unrelated code;
- не добавляй зависимости без явной пользы;
- не меняй публичные контракты без необходимости;
- обнови документацию, если поведение изменилось;
- проверь результат самым коротким релевантным способом;
- в конце сообщи команды проверки и результат.
```

### Шаг 6. Code review после задачи

После реализации:

```text
Сделай глубокое code review активных изменений.
Работай read-only, ничего не меняй.

Ищи только подтвержденные проблемы:
- серьезные регрессии;
- ошибки логики;
- security/privacy risks;
- потерю данных;
- несовместимость API contract;
- места, где не хватает важного теста.

Проверь цепочку:
route/guard -> handler/service -> API contract -> persistence/external services.

Верни список:
- severity;
- file:line;
- доказательство по коду;
- риск;
- минимальный fix;
- какой тест должен это поймать.
```

Если замечания есть:

```text
Составь план исправления только подтвержденных замечаний.
Не принимай вкусовые или спорные комментарии без доказательства.
После исправления снова запусти targeted tests/checks.
```

### Шаг 7. Коммит

Коммит делать после рабочего и проверенного куска:

```text
Проверь git status и diff.
Убедись, что в коммит попадают только изменения этой задачи.
Сделай короткий коммит в imperative style.
В сообщении укажи, что изменилось, без маркетинга и лишних деталей.
```

## Existing Project Workflow

Когда проект уже существует, не начинать с переписывания.

Промпт:

```text
Начни с read-only анализа проекта.

Цель: понять, где живет поведение <описание поведения или бага>.

Действуй так:
1. Посмотри README/AGENTS/документы проекта.
2. Найди вероятную область по структуре.
3. Используй точный поиск, не широкий grep по общим словам.
4. Открой маленькие участки найденных файлов.
5. Найди владельца поведения и связанные контракты.
6. Верни evidence map: path:line, symbol/route/component, краткий смысл, почему важно.

Пока ничего не меняй.
```

После evidence map:

```text
Предложи минимальный change set.

Укажи:
- что именно меняем;
- что не трогаем;
- какой риск;
- какой тест или проверка нужна;
- какой rollback возможен, если что-то пойдет не так.

После моего подтверждения реализуй.
```

## Search Discipline

Правильный порядок поиска:

1. Прочитать `README.md`, `AGENTS.md`, `docs/`, `wiki/`.
2. Посмотреть дерево верхнего уровня.
3. По названию feature/route/component предположить область.
4. Искать узко: route, function name, API path, schema name, UI label.
5. Открывать 20-80 строк вокруг найденного места.
6. Проверить контракты по цепочке.
7. Остановиться, когда владелец поведения найден.

Промпт для делегирования поиска более дешевой модели:

```text
Работай read-only как search subagent.

Найди код, связанный с <задача>.
Не предлагай реализацию и ничего не меняй.

Верни только evidence map:
- path:line;
- symbol/route/component;
- краткий snippet или signature;
- почему это место важно;
- какие связанные файлы стоит открыть основному агенту.
```

## Тестирование

### Когда нужны тесты

Тест нужен, если:

- меняется логика денег, доступа, данных, auth, permissions;
- баг может повториться;
- behavior трудно проверить руками;
- проект уже близко к production;
- рефакторинг затрагивает важное поведение;
- есть интеграция с БД/API/external service.

### Prompt: покрыть главные сценарии

```text
Давай покроем основные сценарии приложения e2e и integration tests.

Сначала сделай глубокое read-only review продукта:
- раздели приложение на фичи;
- выпиши главные пользовательские сценарии;
- оцени, какие сценарии должны быть e2e, какие integration, какие unit;
- не добавляй тесты ради процента покрытия.

Дальше работаем по одной фиче:
1. выбираешь фичу;
2. обсуждаешь, какие сценарии покрыть;
3. пишешь тесты;
4. запускаешь тесты;
5. отдаешь тесты на review с вопросом: можно ли этим тестам верить;
6. исправляешь;
7. коммитишь;
8. переходишь к следующей фиче.

Используй популярные инструменты текущего стека.
```

### Prompt: TDD для бага

```text
Сначала воспроизведи баг тестом.

Правила:
- тест должен падать на текущей реализации;
- тест должен проверять пользовательски важное поведение;
- после падающего теста сделай минимальный fix;
- затем запусти тест и релевантные checks;
- после зеленой проверки можно аккуратно рефакторить.
```

## Refactoring

### Care refactoring

Рефакторинг нужен не для красоты, а чтобы убрать объективно сложный код и сохранить поведение.

Промпт:

```text
Проведи care refactoring review.
Работай сначала read-only.

Найди код, который объективно сложен в поддержке:
- высокая связность;
- дублирование логики;
- слишком длинные функции;
- неочевидные side effects;
- сложные условия;
- смешение UI, state, API и business logic.

Не предлагай косметику.
Для каждого кандидата укажи:
- file:line;
- почему это реально мешает поддержке;
- какой behavior нужно защитить тестом;
- минимальный план рефакторинга;
- риск регрессии.

После согласования покрой важное поведение тестом, затем перепиши код и проверь, что поведение сохранилось.
```

Запускать раз в пару недель или когда проект становится трудно менять.

### UI refactoring

Проблема AI-фронтенда: стили расползаются, компонент переопределяется снаружи, layout и visual state смешиваются.

Промпт:

```text
Проведи UI refactoring review.
Работай read-only.

Ищи:
- inline styles без причины;
- хаотичные className overrides;
- дублирующиеся визуальные состояния;
- компоненты, где layout и visual styling смешаны;
- стили, которые должны жить внутри компонента, но живут у родителя;
- responsive bugs: overflow, clipping, overlap, layout shift.

Верни только actionable findings:
- file:line;
- текущая проблема;
- почему это усложняет поддержку UI;
- минимальный refactor;
- как проверить визуально.
```

## Security

### Перед production

Промпт:

```text
Проведи глубокий security audit перед production.
Работай сначала read-only.

Проверь:
- auth и session handling;
- permissions и role checks;
- ID enumeration в endpoints;
- доступ к чужим данным;
- file uploads;
- secrets и .env;
- CORS/CSRF;
- rate limits;
- payment/webhook validation;
- SQL/ORM query safety;
- logging of private data;
- dependency risks.

Можно запускать независимых review subagents по зонам.
Финально верни:
- confirmed issue;
- severity;
- exploit scenario;
- file:line;
- минимальный fix;
- тест или проверка.
```

### ID enumeration check

```text
Проверь backend endpoints на ID enumeration.

Ищи места, где пользователь может изменить id в URL/body/query и получить, изменить или удалить чужие данные.

Проверь route -> auth guard -> service -> database query.
Верни только подтвержденные проблемы с file:line и минимальным fix.
```

### Supply-chain audit

```text
Проведи read-only supply-chain audit.

Проверь:
- package.json;
- lockfiles;
- installed versions;
- vulnerable package names/versions из новости: <список>;
- transitive dependencies, если lockfile позволяет.

Пока ничего не меняй.
Верни, затронут ли проект, доказательства и план действий.

Если проект затронут:
- предложи safe pinned version;
- убери floating ranges только там, где это нужно;
- не ставь latest без проверки;
- запусти релевантные tests/checks.
```

## Production И Эксплуатация

### Production logs automation

Цель автоматизации - ловить ошибки и сохранять actionable incidents, а не чинить их сразу.

Промпт для настройки:

```text
Создай мониторинг-автоматизацию для production logs.

Сначала изучи, как в этом проекте смотреть production logs.
Затем предложи точную настройку автоматизации.

Режим:
- запуск каждые 30 минут;
- смотреть только последние 30 минут;
- фильтровать ошибки и исключения;
- не сохранять обычные informational logs;
- если найдена ошибка, создать файл errors/<short-error-name>.md.

В error file сохранить:
- время;
- сервис/окружение;
- trace;
- релевантный контекст;
- частоту/количество повторов, если видно;
- ссылки на dashboard/log command, если есть.

Не ищи root cause в этой автоматизации.
Root cause будем разбирать отдельно раз в день.

Если запланированный запуск был пропущен из-за закрытого Codex, при следующем запуске Codex в этот же день выполни мониторинг сразу.
```

Ежедневный разбор:

```text
Открой папку errors/.
Разбери ошибки по одной.

Для каждой:
1. найди root cause;
2. предложи минимальный fix;
3. добавь тест или проверку, если нужно;
4. исправь;
5. проверь;
6. удали error file только после подтвержденного fix.
```

### Load testing перед production

Промпт:

```text
Давай подготовим нагрузочное тестирование backend перед production.

Сначала работай read-only:
- изучи backend, database access, queues, cron, workers, external services;
- найди очевидные bottlenecks под 3-5k RPS;
- предложи список первостепенных правок: проблема, почему важно, как исправить.

Если инфраструктура еще не выбрана:
- сравни подходящие production варианты;
- задай вопросы;
- помоги выбрать минимально достаточный контур.

Если инфраструктура выбрана:
- тестовый контур должен быть похож на production;
- нагрузку запускать не с локальной машины, а с отдельной cloud VM;
- использовать популярный open-source load testing tool.

Сценарии:
- 1k RPS;
- 3k RPS;
- 5k RPS;
- 10k RPS только как верхняя проверка, если это не ведет к лишним расходам.

Во время теста:
- смотреть logs;
- сохранять найденные баги в notes/<note.md>;
- после теста разложить их в tasks/<task.md>.

Финально напиши production runbook:
- минимальная стартовая infrastructure;
- как вертикально масштабировать DB;
- как горизонтально масштабировать backend;
- какие метрики смотреть;
- когда масштабироваться.
```

## Long-Run Agent Workflow

Для длинной очереди задач можно запускать автономный цикл, но только если уже есть хороший `PRD.md`, `TASKS.md`, тесты и лимиты.

```text
Работай по TASKS.md.

Цикл:
1. Возьми первую открытую задачу.
2. Перед реализацией запусти risk review subagent:
   - подводные камни;
   - edge cases;
   - проблемные места реализации;
   - какие тесты нужны.
3. Реализуй минимальный change set.
4. Запусти релевантные tests/checks.
5. Запусти independent code review.
6. Исправь подтвержденные замечания.
7. Повтори review, если были серьезные правки.
8. Сделай commit.
9. Отметь задачу выполненной.
10. Перейди к следующей.

Не продолжай, если:
- тесты не проходят;
- review нашел критическую проблему;
- задача требует продуктового решения;
- нужно разрушительное действие;
- есть риск затронуть чужие изменения.
```

## Automation Ideas

### 1. Project bootstrap automation

Для `vibe-project-installer` уже есть полезная идея: создавать проект из шаблона с выбранными surfaces и hosting posture, затем писать bootstrap plan в README.

Что можно добавить позже:

- examples matrix: `webapp`, `website`, `backend`, `full-stack`, `mobile`;
- smoke tests для разных аргументов;
- проверку, что upstream remote удален;
- генерацию начального `PRD.md`/`TASKS.md` skeleton.

### 2. Review skill

Сделать локальный skill `/loop-code-review`, который:

- запускает несколько независимых ревью;
- собирает только actionable findings;
- отбрасывает вкусовые замечания;
- требует file:line и доказательство;
- завершает цикл, когда нет важных замечаний.

### 3. Search subagent skill

Сделать skill `/code-search-map`:

- read-only;
- принимает feature/bug;
- возвращает evidence map;
- не предлагает реализацию;
- экономит основной контекст.

### 4. Production log monitor

Codex automation:

- каждые 30 минут;
- легкая модель;
- пишет только `errors/*.md`;
- не чинит;
- догоняет пропущенный запуск в тот же день.

### 5. Daily error triage

Отдельная automation или ручной daily task:

- читает `errors/`;
- выбирает одну ошибку;
- делает root-cause analysis;
- чинит;
- проверяет;
- удаляет файл ошибки после fix.

### 6. Dependency incident scanner

Read-only automation по новостям supply-chain:

- список репозиториев;
- package/lockfile inventory;
- affected/not affected report;
- никаких правок без подтверждения.

## Готовые Промпты По Ситуациям

### Начать новый веб-продукт

```text
Я хочу сделать веб-продукт: <описание>.

Сначала не пиши код.
Проведи PRD-интервью: задавай вопросы по одному с вариантами ответа.
Когда вопросов не останется, создай PRD.md без открытых вопросов.
После этого остановись и кратко скажи, что готово.
```

### Разбить PRD на задачи

```text
Прочитай PRD.md.
Создай TASKS.md с этапами, задачами, зависимостями и проверками.
Каждая задача должна быть маленькой, проверяемой и давать рабочий результат.
Не добавляй функциональность, которой нет в PRD.
```

### Реализовать одну задачу

```text
Возьми первую открытую задачу из TASKS.md.
Сначала read-only: найди связанные файлы и риски.
Потом предложи минимальный план.
После согласования реализуй, проверь и обнови TASKS.md.
```

### Исправить баг

```text
Найди root cause бага: <описание>.

Сначала ничего не меняй.
Прочитай логи/ошибку/связанный код.
Покажи цепочку причины.
Если возможно, сначала добавь падающий тест.
Потом сделай минимальный fix и проверь.
```

### Сделать review

```text
Сделай read-only code review активных изменений.
Ищи только подтвержденные серьезные проблемы.
Верни severity, file:line, доказательство, риск, минимальный fix и тест.
Не пиши вкусовые замечания.
```

### Подготовить к production

```text
Проведи production readiness review.

Проверь:
- security;
- secrets;
- auth/permissions;
- DB migrations;
- logging;
- backups;
- monitoring;
- error handling;
- deployment path;
- rollback;
- smoke tests.

Верни блокеры, риски и минимальный план выхода.
```

## Как Работать Пользователю

### Каждый день

1. Начинать с чистого `git status`.
2. Выбирать одну задачу из `TASKS.md`.
3. Давать агенту контекст задачи, а не весь продукт заново.
4. Требовать read-only анализ перед изменениями.
5. После изменений требовать проверку и review.
6. Коммитить рабочий кусок.
7. Обновлять `TASKS.md` и `wiki`, если появилось долговременное знание.

### Для сложной фичи

1. Отдельное окно: обсудить продуктовые edge cases.
2. То же окно: составить план.
3. Реализация.
4. Отдельное окно или subagent: review.
5. Исправления.
6. Тесты.
7. Коммит.

### Для безопасности

1. Новый чистый контекст.
2. Read-only audit.
3. Только подтвержденные findings.
4. Fix по одному.
5. Повторный audit в новом контексте.
6. Повторять, пока не останутся только accepted risks.

### Для production

1. Production readiness review.
2. Security audit.
3. Load test plan.
4. Monitoring logs automation.
5. Smoke test после deploy.
6. Daily errors triage.

## Что Не Делать

- Не начинать большой продукт сразу с кода.
- Не делать много задач в одном контексте без checkpoints.
- Не принимать review без file:line и доказательства.
- Не добавлять MCP/skills/dependencies на всякий случай.
- Не заменять точный поиск по коду абстрактной памятью.
- Не хардкодить секреты.
- Не пушить в upstream template.
- Не делать refactor без проверки поведения.
- Не тащить mobile, Docker, EAS, cloud deploy, если они не нужны сейчас.
- Не просить агента чинить production logs прямо в monitoring automation.
